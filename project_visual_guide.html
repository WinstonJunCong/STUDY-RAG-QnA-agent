"Retrieval-Augmented Generation" mapped to standard dev concepts.

1. Chunking (index_builder.py)

The Problem: You have a massive 10MB text log, but an LLM API request has a strict payload size limit (called the "context window"), which takes exactly ~4000 tokens.

The Solution: The SentenceSplitter takes strings and cuts them into 512-word chunks. It slightly overlaps the chunks so you don't cut a sentence or thought perfectly in half, losing context.

2. Embeddings & VectorDB

The Problem: How do you run a semantic search for "dogs" without writing a massive SQL LIKE '%犬%' OR '%puppy%' query?

The Solution: We pass every chunk object to an Embedding Model (`BGE-small`). This model acts like a hash function that returns an array of floats [0.1, -0.4, 0.9...] describing the "meaning". We save this chunk+float array to ChromaDB (a VectorDB). Searches are just computing cosine similarity (distance) between float arrays!

3. Q&A Generation (qa.py)

The Workflow: User queries something. We embed their prompt into a float array. We search ChromaDB for the 5 most similar chunks. We pull those out.

The Prompt Injection: We literally just concatenate strings: "Answer the question using this context: " + chunks + " \n\n Question: " + user_query, and send this single massive string to a local LLM API (Ollama).
