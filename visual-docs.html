The architecture is divided into three main stages: Ingestion Pipeline, Storage Layer, and Query Pipeline. Each stage is designed to handle different aspects of the RAG workflow.

4.1 Document Parsing: Unstructured.io vs Simple Chunking

The Problem with Simple Splitting

Traditional character-count splitting can break document structure:

Simple Split at 100 chars:
Chunk 1: "Pricing & Plans FAQ\n\n| Plan | Price | Age
Chunk 2: "nts |\n|----------|------------|--------|"
Chunk 3: "...\n\nResult: Table structure destroyed"
Unstructured.io Result:
Chunk: "Pricing & Plans FAQ\n\n| Plan | Price | Price |\n|----------|------------|----------|\n| Starter | $19/mo | Up to 3|\n| Growth | $49/mo | Up to 15|\n| Enterprise| Custom | Unlimited|\n\nAnnual billing gives a 20% discount."
Simple Splitting
Table handling✗ Splits rows
Q&A pairs✗ May separate Q from A
Code blocks✗ May break mid-line
Headings✗ Ignored
Lists✗ May split items
Context preservation✗ Loses relationships
Unstructured.io
Table handling✓ Preserves structure
Q&A pairs✓ Keeps together
Code blocks✓ Intact
Headings✓ Used as chunk boundaries
Lists✓ Keeps related items
Context preservation✓ Maintains structure

⚙️ Parameter Rationale

5.1 Chunk Sizes: [2000, 1000, 350]

CHUNK SIZE RATIONALE
MAX_CHARS: 2000
Hard ceiling per chunk. LLM context is 8K tokens. With 6 chunks of ~500 tokens each, we use ~3K tokens, leaving room for prompt.
SOFT_LIMIT: 1000
Half of MAX. Encourages splitting at logical midpoints (paragraphs, sections) rather than mid-sentence.
MIN_CHARS: 350
Merge fragments smaller than this. Smaller chunks lose meaning (e.g., just "Pricing & Plans"). Merge to preserve context.

Example: MIN_CHARS Impact

Without MIN_CHARS:
Chunk: "Pricing"
Chunk: "& Plans"
Chunk: "FAQ"
With MIN_CHARS (350):
Chunk: "Pricing & Plans FAQ\n\nQ: What plans does NovaDesk offer?\nA: We offer Starter, Growth, and Enterprise."
Parameter Value Reasoning
MAX_CHARS 2000 LLM context headroom
SOFT_LIMIT 1000 Semantic splits at paragraph breaks
MIN_CHARS 350 Merge small fragments to preserve context

5.2 MMR Lambda = 0.7

What is MMR?

Maximal Marginal Relevance balances two goals:

  • Relevance - Return chunks similar to the query
  • Diversity - Don't return all similar chunks (avoid redundancy)

Formula

mmr_score.py 
MMR Score = λ × similarity(query, doc) - (1-λ) × max_similarity(doc, other_docs)

Where λ (lambda) controls the balance:
- λ = 1.0 → Pure relevance (might return duplicates)
- λ = 0.0 → Pure diversity (might return irrelevant)

Testing Results

Lambda Q1 Q2 Q3 Q9 Avg Notes
0.6 100 100 100 100 92.2% Baseline
0.7 100 100 100 100 94.0% Best balance
0.8 95 100 100 100 89.5% Too much diversity

Why TOP_K = 6?

TOP_K Pros Cons Result
3 Fast, focused Might miss context ✗ Lower scores
6 Good balance Slightly slower ✓ Best
12 Comprehensive LLM overwhelmed ✗ Lower scores

Why BM25_TOP_K = 8?

BM25 candidates are filtered by RRF. Start with more candidates (8) to ensure keyword matches aren't missed, then RRF ranks and selects top 6.

6.1 RELEVANCE GATE

The Problem

Question:
"Can I import tickets from Zendesk?"
Retrieved Chunks:
  • "Connect Slack: Settings → Integrations → Slack..."
  • "API: Create, update, close tickets..."
  • "Export: Reports → Export → CSV..."
LLM Response (Hallucinated!):
"You can connect Slack to NovaDesk via Settings. The API allows creating and updating tickets. You can export data as CSV. Regarding Zendesk import..."

❌ Hallucination - LLM combined unrelated topics into an answer.

The Solution

agent/qa.py 
specific_terms = {'zendesk', 'hipaa', 'soc', 'slack', 'api', 'import', 'export', 'gdpr', 'sso', 'saml', 'oauth', 'webhook', 'zapier'}

question_lower = question.lower()
question_words = set(re.findall(r'\b[a-z]{4,}\b', question_lower))
question_specific = question_words & specific_terms

if question_specific:
    chunk_texts = [n.text.lower() for n in nodes]
    all_found = all(
        any(term in chunk for chunk in chunk_texts)
        for term in question_specific
    )
    if not all_found:
        return {"answer": "I couldn't find that in the provided documents."}
Why Code Instead of Prompt?
Prompt RuleFlexible
❌ LLM sometimes ignores it
Code CheckDeterministic
✅ Always reliable

6.2 PRICING QUESTIONS Rule

The Problem

Question:
"How much does Growth plan cost?"
Expected:
"$49/mo, up to 15 agents, 20% off annually"
LLM Response:
"$49 per agent per month"

❌ Missing agent limit and annual discount.

The Solution

qa.py - PRICING QUESTIONS 
"PRICING QUESTIONS: For questions about plan costs, extract ALL pricing details:
  - Monthly price per agent
  - Agent seat limits
  - Annual/discount pricing (e.g. 20% off annually)
  - Trial information

  Example correct: '$49/agent/month, up to 15 agents, 20% off annually'
  Example WRONG:   '$49/agent/month' (missing agent limit and discount)"

6.3 CONFLICT CITATION Rule

The Problem (Q9: SLA Conflict)

Question:
"What is SLA first response time for Urgent tickets?"
Doc 11 (current):
"Urgent SLA = 1 hour"
Doc 12 (legacy):
"Urgent SLA = 30 minutes"
❌ WRONG LLM Response:
"According to doc 11, the SLA is 1 hour. Please verify current settings."

❌ Issues:

  • Ignored doc 12 (legacy 30-minute value)
  • Hedged with "verify settings" instead of stating both values
  • Didn't explain why there's a conflict

The Solution

qa.py - CONFLICT CITATION 
"CONFLICT DETECTION: Only flag a conflict if the SAME specific fact has
different values in different sources.
  - REAL conflict:   Source A says first response = 30 min, Source B says 1 hour for the same ticket priority → Flag it.
  - FALSE conflict:  Source A says Starter has 3 agents, Source B says Growth has 15 → Different plans, not a conflict.

CONFLICT CITATION (CRITICAL): If multiple sources provide DIFFERENT values for the SAME metric, you MUST include the ACTUAL VALUE from EACH source.
  - WRONG: 'According to [doc 12], the urgent SLA is 30 min. However, [doc 11] shows the current default is 1 hour.'
  - CORRECT: 'According to [doc 11], the urgent SLA first response is 1 hour. According to [doc 12], the urgent SLA first response is 30 minutes (legacy).'
  - Do NOT say 'verify settings' without stating the actual conflicting values."

6.4 YES/NO Framing

The Problem

Question:
"Can I use AI Bot on Starter plan?"
❌ WRONG Response:
"The AI Bot is not available on the Starter plan, but it is available on Growth and Enterprise plans."

❌ Problem: "but..." makes it sound like there IS a way on Starter. The answer is clearly No.

The Solution

qa.py - YES/NO QUESTIONS 
"YES/NO QUESTIONS: If the direct answer is No, your FIRST word must be 'No.'
  - Correct:   'No, the AI Bot is only available on Growth and Enterprise plans.'
  - Incorrect: 'The AI Bot is not available on the Starter plan, but...'
  - Incorrect: 'Yes, but only on Growth and Enterprise.'
  - SDK/feature exists but no standalone product: Lead with what does NOT exist, then mention what does."

6.5 NOT FOUND Handling

qa.py - NOT FOUND 
"NOT FOUND: If the answer is not in the sources, say 'I couldn't find that in the provided documents.' — one sentence, nothing else.
Do NOT suggest alternatives, workarounds, or related features."

Why "nothing else"?

Response User Feels
"I couldn't find that. You might try checking our website..." ✗ Confused, frustrated
"I couldn't find that." ✓ Clear, honest

🧪 Testing Strategy

7.1 Why LLM-as-Judge?

Manual Testing
  • 17 questions × 2 min = 34 min
  • ❌ Variable scoring
  • ❌ Inconsistent
  • ❌ Not scalable
LLM-as-Judge
  • 17 questions × 5 sec = 85 sec
  • ✅ Consistent scoring
  • ✅ Deterministic
  • ✅ Scalable

⚠️ Failure Cases Deep Dives

Analysis of 4 key failure cases and their resolutions.

1 Case 1: Q9 - SLA Conflict (1 hour vs 30 minutes)
The Question
"What is the SLA first response time for an Urgent ticket?"
The Documents
Doc 11 (11_automations_sla_faq.md):
## Default SLA policy: | Priority | First Response | Resolution | |----------|---------------|------------| | Urgent | 1 hour | 4 hours |
Doc 12 (12_sla_legacy_overlap.md):
## Default SLA Times (v3.x — verify in your current settings) | Priority | First Response | Resolution | |----------|---------------|------------| | Urgent | 30 minutes | 2 hours |
The Problem
❌ Initial LLM Response:
"The SLA first response for Urgent tickets is 1 hour according to doc 11. Please verify the current settings in your NovaDesk."
  • Ignored doc 12 (legacy 30-minute value)
  • Hedged with "verify settings"
  • Didn't explain why there's a conflict
  • Both values should have been stated
Root Cause Analysis
The LLM was trained to give "helpful" answers. Saying "please verify settings" feels helpful but doesn't actually answer the question with the information available.
The Fix
Added explicit CONFLICT CITATION rule requiring both values to be stated.
✅ Result After Fix
"The SLA first response time for Urgent tickets is 1 hour according to the current defaults (Automations & SLA FAQ). However, the legacy documentation shows 30 minutes for the same priority. Please verify your current settings in your NovaDesk instance. (Source: 11_automations_sla_faq.md, 12_sla_legacy_overlap.md)"
Score: 100 (Pass)

✅ Key Takeaways

01
Hybrid Retrieval > Single Method
✗ Vector-only: Misses exact keyword matches (Zendesk, HIPAA)
✗ BM25-only: Misses semantic variations (cheapest → lowest price)
✓ Hybrid: Captures both
02
Chunking Quality = Answer Quality
✗ Simple splitting: Destroys tables, Q&A pairs, code blocks
✗ Semantic chunking needed for structure
✓ Unstructured.io preserves structure → Better context → Better answers
03
Prompts Need Examples
✗ Rule only: "Extract all relevant details."
✗ LLM learns better from negative examples
✓ With examples: "Example correct: '...' Example WRONG: '...'"
04
Code-Level Safety Nets
✗ LLM sometimes ignores prompt rules
✗ Prompts can be lenient
✓ Code checks are deterministic and reliable
Use code for edge cases (missing topics, hallucinations)
05
Iterate Based on Failures
✗ Don't over-engineer upfront
✗ Guessing doesn't work
✓ Every failure → Analyze → New rule or parameter tweak
Let real failures guide improvements
06
Testing Enables Iteration
✗ Manual testing: Slow, inconsistent, not scalable
✗ Time-consuming review process
✓ LLM-as-judge: Fast, consistent, enables rapid iteration
